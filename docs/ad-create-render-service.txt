AD CREATE & RENDER SERVICE — END-TO-END
============================================

Purpose
-------
Turn operator selections from the Generate Ads wizard into a queue of
viable ad combinations, then drain that queue through Puppeteer into
rendered creatives on Cloudinary. Queue-first architecture: every
combination is persisted up front; render is a separate selection
pass that drains the inventory.

Architecture shift landed 2026-05-11. Previous model built an
in-memory cartesian, sliced to a cap, dropped the rest. New model
persists everything and renders the top N per run.

Doc maintenance
---------------
Update this doc whenever any of the services below change:
  - campaignAdsGenerationService.js  (seeding, scoring, cartesian, dedup)
  - layoutInputService.js            (context load, derivation, assembleInput)
  - templateRegistry.js              (slot/style bindings, contrast guard)
  - renderService.js                 (Puppeteer, Cloudinary, persistence)
  - routes/ads.js                    (endpoints, run orchestration)
The deep-dive sections (Stage 6a, 6b, 7) are the most likely to drift —
keep field lists and binding rules in sync with the code.


High-level flow
---------------
POST /api/ads/generate
  ↓
1. expandWizardJob     — seeds × templates × ratios × variants → bulk
                         insertMany of Ad docs with status='queued'
  ↓
2. selectAdsForRun     — pick top N queued by readinessScore desc
  ↓
3. mark rendering      — selected Ads flipped to status='rendering',
                         campaignRunId stamped
  ↓
4. CampaignRun.create  — batch tracker doc
  ↓
5. 202 response        — frontend redirects, starts polling
  ↓
6. runRenderLoop       — setImmediate; concurrency-bounded dispatch
   ↓
   renderOne(adId) for each picked Ad:
     ↓
     renderCreative(req)   — derive → validate → render → upload → persist
       success: Ad → status='draft' + render output backfilled
       failure: Ad → status='failed' + renderError diagnostics


Stage 1 — expandWizardJob (services/campaignAdsGenerationService.js)
--------------------------------------------------------------------
Inputs:
  campaignId, productIds[], mediaIds[], templateIds[], cta, urlParams,
  requestedBy

Seed expansion rules (per operator pick):

  1. No picks (brand-only):
     → top BRAND_ONLY_MEDIA_LIMIT (25) brand_match media for the
       brand, ranked by the blended engagement+adSuitability score
       used in readinessScoreFor. Emit ugc seeds with productId:null
       and matchTier='brand_only'.

  2. mediaIds (library entry — operator picked a specific post):
     The post is ALWAYS the visual hero — seedsFromMedia never emits
     variantKind='product_image'. Outcome decides which products the
     post pairs with:

       Case A — at least one refined product is product_match
         → emit one ugc seed per matched product
           { productId, mediaId, matchTier='product_match',
             variantKind='ugc', platformStats, fileType,
             suitabilityScore }

       Case B — only product_category outcomes (no product_match)
         → expand to top-K products in the matched categories,
           ranked by productPopularityScore. Each becomes a seed
           with matchTier='product_category'. K = EXPANSION_PRODUCTS_
           PER_POST (default 25).

       Case C — brand_match (no product or category signal)
         → expand to top-K products in the brand's full catalog,
           ranked by productPopularityScore. Each becomes a seed
           with matchTier='brand_match'.

     Cross-product pairings in Cases B + C are SYNTHETIC — the post
     never showed the paired product. Tier inheritance reflects
     this: the seed inherits the post's outcome tier, not
     product_match. A brand-only post expanded to 25 products
     drains AFTER actual product_match UGC seeds.

     productPopularityScore composite:
       log10(matchedMedia.length + 1) / 2          (UGC popularity)
     + (rating/5) × log10(reviewCount + 1) / 3     (review strength)
       capped at 1.0
     Primary signal is matchedMedia count (how often the product has
     been featured in UGC); review strength is a tiebreaker for
     products with no UGC matches yet.

     Carousel posts (multi-asset): currently flattened to first child
     by postSyncService — carousel-aware expansion is backlogged.

  3. productIds (catalog entry — operator picked a product):
     → Read CatalogProduct.matchedMedia[] for the per-product tiers
       (product_match + product_category):

         CatalogProduct.matchedMedia
            → ugc seed per entry, matchTier from the array

     → Union in ALL brand_match media for the brand (queried from
       ProductMatchArtifact — not denormalized on CatalogProduct
       since brand_match isn't per-product):

         brand_match PMAs for brandId
            → ugc seed with productId tagged, matchTier='brand_match'

     → PLUS ONE product_image seed per product —
       uses the catalog product's hero Media doc as the visual slot,
       variantKind='product_image', matchTier='product_match' (the
       product IS the SKU here)

  4. Both productIds + mediaIds: union, then dedupeSeeds by
     (productId|null, mediaId, variantKind).

Cartesian expansion:
  for each seed:
    for each (template, aspectRatio) cell in
        allowedTemplates × intersect(template.ratios, SHIPPING_RATIOS):
      for each paletteSource in ['media', 'brand']:
        payload = {
          brandId, campaignId, mediaId, productId,
          template, aspectRatio, campaignKind,
          matchTier, variantKind, paletteSource,
          readinessScore: computed (see below),
          status: 'queued',
          identityDigest: sha256(canonical-form identity
                                  — includes paletteSource),
          cta*, queuedAt, generatedAt
        }
        payloads.push(payload)

  SHIPPING_RATIOS = { '1:1', '9:16', '16:9' }   — V1 only

  paletteSource doubles the cartesian. 'media' draws style bindings
  from the hero media's extracted palette (palette_dominant /
  palette_vibrant etc.). 'brand' overrides those slots with
  Brand.primaryColor / accentColor / secondaryColor at assembleInput
  time. Templates' style_bindings read media.palette_* unchanged —
  the value swap is invisible to the template JSON.

After cartesian, payloads sorted by readinessScore desc and trimmed
to MAX_ADS_PER_GENERATION_RUN (env, default 200). Bounds per-run
growth on large catalogs. Re-running the wizard queues the next
batch (identityDigest dedup catches duplicates at insert).

Bulk insert:
  Ad.insertMany(payloads, { ordered: false })

  { ordered: false } means dup-key errors on the
  (campaignId, identityDigest) unique index don't abort the batch.
  Idempotent: re-running with the same picks queues only new combos.

Readiness score computation:
  TIER_WEIGHTS = {
    product_match:    1.0,
    product_category: 0.8,
    brand_match:      0.6,
    brand_only:       0.5
  }

  variantKind='product_image':
    score = tierWeight × 0.7  (constant — catalog hero quality)

  variantKind='ugc' (image OR video — equal participation):
    quality   = 0.6 × engagementScore  +  0.4 × adSuitability.score
    score     = tierWeight × quality

  engagementScore (per Media.platformStats):
    raw     = likes + 2×comments + 2×saves + 3×shares
              (saves and shares are higher-intent than likes;
               comments express deeper engagement)
    score   = min(1, log10(raw + 1) / 4)
              (10 ≈ 0.26, 100 ≈ 0.50, 1000 ≈ 0.75, 10000 ≈ 1.0;
               viral posts don't dwarf the rest of the queue)
    null when raw <= 0 → caller blends a 0.5 default

  Null-side fallback: if either engagement or adSuitability is null,
  the missing side contributes a 0.5 default. Avoids penalizing
  single-signal media (e.g. freshly-ingested posts with adSuitability
  set but no insights yet).

  Effect on drain order: within a tier, highest-engagement posts
  surface first. Across tiers, tier weight dominates — a brand_match
  post with viral engagement (0.6 × 1.0 = 0.60) still ranks below a
  weak product_match (1.0 × 0.5 = 0.50)... wait, equal. Actually
  brand_match-viral (0.60) edges out product_match-weak (0.50). The
  blended math is intentionally permissive at the boundaries so a
  truly outstanding lower-tier post can outrank a mediocre higher-tier
  post.

Returns:
  {
    campaignId, brandId, campaignKind,
    queuedCount,    — total Ad{status:'queued'} for the campaign
    newlyQueued,    — inserted in this call
    alreadyQueued,  — combos already in the queue (insert no-op)
    newAdIds        — ObjectIds of the docs just inserted
  }


Stage 2 — selectAdsForRun
-------------------------
Inputs:  { campaignId, limit }
Output:  Ad IDs (strings) to render this batch

Query:
  Ad.find({ campaignId, status: 'queued' })
    .sort({ readinessScore: -1, queuedAt: 1 })
    .limit(limit)

readinessScore: -1 puts null videos last automatically (Mongo desc
sort).
queuedAt: 1 as secondary key gives FIFO among equal-readiness ads.

MAX_CREATIVES_PER_RUN (env, default 6) is the limit. Bounded because
Puppeteer + Cloudinary OOMs Chromium past ~2 in-flight; six is
"two templates × three ratios for one seed" baseline.

Multiple runs of the same campaign DON'T re-queue ads — selection
just keeps draining the existing queue.


Stage 3 — Mark selected Ads as rendering
----------------------------------------
Ad.updateMany(
  { _id: { $in: adIds } },
  { $set: { campaignRunId: runId, status: 'rendering' } }
)

Up-front so a server restart leaves the docs in a recognizable
in-flight state (operator can see "rendering" on the Ads page)
rather than orphaned-but-queued.


Stage 4 — CampaignRun
---------------------
A batch tracker doc, models/CampaignRun.js:
  runId, brandId, campaignId, campaignKind,
  total, succeeded, skipped, failed,
  status: 'running' | 'done' | 'failed',
  errors: [{ index, stage, template, aspectRatio, mediaId, productId, message }],
  startedAt, completedAt

Used by GET /api/ads/runs/:runId for polling progress from the
frontend after the 202.


Stage 5 — runRenderLoop (routes/ads.js)
---------------------------------------
fire-and-forget after the 202 flushes. Iterates picked Ad IDs with
RENDER_CONCURRENCY (env, default 2) in flight. Each Ad goes through
renderOne(run, job, adId, index, renderToken).

renderOne flow:
  1. Fetch Ad doc by adId.
  2. Shape into the creative descriptor renderService expects.
  3. Call renderCreative({adId, campaignId, campaignRunId, brandId,
                          campaignKind, creative, cta, authToken}).
  4. On 'success' → CampaignRun.succeeded++
  5. On 'skipped' → CampaignRun.skipped++ (template validation failed)
  6. On 'failed' → CampaignRun.failed++ + push to errors[]
                  + Ad.updateOne(status='failed', renderError stamped,
                                  renderAttempts++)
  7. On crash → same as failed with stage='crash'

When the queue drains, CampaignRun.status → 'done'.


Stage 6 — renderCreative (services/renderService.js)
----------------------------------------------------
Five-stage pipeline. Each stage labels its errors so CampaignRun.errors[]
records which step failed.

  derive   → buildLayoutInput(mediaId, template, aspectRatio, options)
             Re-uses cached LayoutInputArtifact when schemaVersion
             matches; otherwise runs Gemini-backed copy derivation +
             slot resolution. options threads campaignKind + cta so
             the prompt flips to brand-mode copy or composes the
             right CTA imperative.

  validate → templateRegistry.validateInputAgainstTemplate(input, template)
             Pre-flight check against the template's normalized
             required_all_of / required_any_of / minCount rules.
             Failures return status:'skipped' (doesn't count as
             render failure — the inputs just couldn't satisfy the
             template).

  render   → renderStage({ layoutInputArtifactId, template, ratio,
                            mediaId, brandId, authToken, renderMode })
             Spins up Puppeteer, navigates to
             FRONTEND_URL/ads.html?...&renderMode=1 (with the auth
             token in cookie/URL so requireAuth resolves req.user),
             waits for window.__tpRenderReady === true, screenshots
             #tpStage. renderMode 'video-overlay' produces a
             transparent-slot overlay PNG; 'static' produces the full
             rendered image.

  upload   → uploadStage(renderOutput, ctx)
             Cloudinary upload, folder ads/{brandId}/{campaignId}.
             publicId: {ratio}-{template}-{shortMedia}-{shortIdentityDigest}
             (deterministic; overwrite=true so the same identityDigest
             always overwrites the prior asset).
             Returns { renderUrl, cloudinaryPublicId, width, height,
                      bytes, durationMs, posterUrl? }.

  composeVideoOutput (video branch only):
             Chains Cloudinary transforms to overlay the PNG on the
             cropped source video. Builds the slot-aware c_crop URL
             via buildVideoCompositeUrl (services/videoCompositeService.js).
             Failure falls through to the static PNG.

  persist  → Ad.findByIdAndUpdate(adId, {
               renderUrl, posterUrl, cloudinaryPublicId,
               width, height, bytes, durationMs,
               kind, copy: extractCopySnapshot(input),
               status: 'draft',
               renderedAt: new Date()
             }, { $inc: { renderAttempts: 1 } })
             Same doc that was queued — updates in place. The Ad's
             identity fields (campaignId, mediaId, productId, template,
             aspectRatio, matchTier, variantKind, identityDigest) stay
             as they were at queue time.

Total per render: ~3-15s (derive 1-5s, render 2-8s, upload 1-3s,
                          persist <100ms)


Stage 6a — Layout input assembly (services/layoutInputService.js)
-----------------------------------------------------------------
The derive substage above resolves to a single canonical input
document. Lifecycle:

  cache lookup → loadContext → derivation (Gemini) → assembleInput
  → upsert LayoutInputArtifact

Cache key:
  (mediaId, template, aspectRatio, productId, variantKind)
  + INPUT_SCHEMA_VERSION (currently 4.0)
  A mismatch on schemaVersion invalidates the cached input even if the
  inputs are identical. Bump the version after structural shape
  changes; leave it alone for content-only changes.

loadContext (lines 370-475)
---------------------------
Pulls every read source the prompt and slot adapter could need:

  Media                       subjects, caption, creatorName,
                              creatorHandle, fileType, fileUrl,
                              adSuitability, platformStats, metadata,
                              classification, latestArtifacts
  DetectionArtifact           primarySubjectDesc, text[], background
                              {palette, description, style, lighting,
                              setting}, transcript
  CropArtifact                smartCrops{5:4, 1:1, 4:5}, winners
                              (per-ratio judge picks)
  ExtendedCropArtifact        9:16 / 1.91:1 AI-extended candidates
                              (skipped on catalog product Media)
  OverlayZoneArtifact         brightness grid, restriction zones
                              (overlay-mode templates only)
  ProductMatchArtifact        outcome, identification.details
                              (productName, brand, price, imageUrl,
                              productUrl, productReviews, reviewSummary),
                              brandReviews, recommendedProducts
                              (hydrated via productMatchHydration)
  CatalogProduct              full row + hero Media + its CropArtifact
                              winners → ctx.productHero
                              (only when options.productId is set)
  Brand                       name, tagline, logoUrl, primaryColor,
                              secondaryColor, accentColor, fontFamily,
                              tone[], demographics[], brandReviews
  Category siblings           via CatalogProduct.categoryRef, cap 12
                              → ctx.categoryPool

Product override path (when options.productId is set):
  - No prior PMA   → synthesizeMatchFromCatalogProduct(cp) builds a
                     stub match from the catalog row
  - Prior PMA      → buildIdentificationFromCatalogProduct(cp)
                     overrides match.identification with catalog
                     details (keeps categoryId / brandReviews /
                     recommendedProducts)
  Catalog hero is loaded into ctx.productHero so the product.image
  slot serves the judge-arbitrated crop, not the raw Shopify URL.

For overlay-mode templates (testimonial_overlay / product_overlay):
  computeOverlayPlacement() runs BEFORE derivation so the prompt can
  include per-element box budgets. extractOverlayBoxes() returns
  { headline: {w_px,h_px}, cta: {w_px,h_px}, quote: {w_px,h_px} }.

Derivation prompt (buildDerivationPrompt, lines 556-799)
--------------------------------------------------------
Up-front signal block (always emitted):
  - Template + aspect ratio + intent
  - Match type line — product_match | product_category | brand_match
    + guidance on how specific feature claims can be (exact SKU vs
    category vs brand-only)
  - Caller tone hint when present

Campaign-intent section (kind='brand' or 'promotional' only):
  - brand: anchor headlines on Brand.tagline; LLM produces
    tone-aware variants matching Brand.tone[]
  - promotional: operator-supplied offer details (discountType,
    discountValue, code, window, giveaway, notes); auto-injects
    urgency phrasing when endsAt is within 7 days

Variant-aware sections:

  BRAND section          when outcome=brand_match — tagline, tone[],
                         scene, sentiment summary. Prompt explicitly
                         forbids inventing product names.
  PRODUCT section        otherwise — productName, brand, category,
                         price, description, visible text from
                         DetectionArtifact.text[]
  SOCIAL CONTEXT         only when variantKind !== 'product_image'.
                         Caption, creator handle, platformStats
                         (likes/comments/shares/saves/views),
                         transcript (video).
                         Skipped for product_image so the prompt
                         stays product-focused.
  REVIEW SIGNAL          real productReviews.quotes (cap 5) →
                         brandReviews.quotes (brand_match) →
                         LLM fallback. Prompt enforces that real
                         review quotes are preferred over invented
                         ones.
  SCENE CONTEXT          background palette / setting / lighting /
                         style from DetectionArtifact.background
  BRAND KEY PERSONAS     Brand.demographics[] surfaced as plausible
                         quote authors
  TASK rules             - per-template quote count + length caps
                           (ugc_split_screen / review_collage:
                           ≤90 chars / 10-14 words; others:
                           ≤120 chars / 12-18 words)
                         - slot-aware character budgets via
                           computeSlotBudgets() — derived from the
                           template's canvas variant
                         - split-headline rules
                           (headline_lead + headline_main)
                         - overlay box budgets when precomputed
                         - brand-mode vs. product-mode copy mode

Gemini call:
  Model:           GEMINI_MODEL (default gemini-2.5-pro)
  Temperature:     0.3
  maxOutputTokens: 3000
  thinkingBudget:  1024
  Schema-locked:   DERIVATION_SCHEMA (lines 175-230)
                   { quotes[], short_benefits[], badges[],
                     copy: { headline, headline_lead, headline_main,
                             subheadline, eyebrow, highlight_text },
                     cta:  { text, subtext, offer_text },
                     trusted_by_text, tone[], theme_style,
                     background_style, emphasis }

On timeout / parse failure → fallbackDerivation() returns a generic
copy skeleton so the ad still renders, just with weaker copy.

assembleInput (lines 876-1214)
------------------------------
Merges derivation + context into the canonical RS Social Proof JSON.
Key behaviors:

  Hero / secondary / creator media picking
    pickHeroMedia, pickSecondaryMedia, pickCreatorMedia choose the
    smartCrop winner closest to the template's media slot ratio.
    Fall through extended crops → fileUrl when nothing matches.

  Quote priority (lines 910-941)
    1. Real productReviews.quotes (or brandReviews when brand_match)
    2. LLM-authored quotes from derivation.quotes
    3. Synthesis from reviewSummary.summary first sentence
    First survivor wins social_proof.primary_quote; rest fill
    secondary_quotes[].

  Variant suppression (variantKind === 'product_image')
    creator block      → {}
    ugc block          → {}
    performance.engagement → {}
    performance.metrics    → []
    Catalog product images have no creator, no post, no engagement
    data — the ad reads as a static product card rather than UGC
    social proof.

Where layout actually lives — two schemas, one renderer
-------------------------------------------------------
Template geometry lives in TWO schemas today:

  canvas.v1.json (canvas-mode templates):
    testimonial_spotlight, ugc_split_screen, review_collage,
    results_proof, creator_endorsement
    Declares fixed rect per ratio: zones[] with id, kind, slot,
    rect{x,y,w,h}, style_variant, layer, padding, radius. Renderer
    reads canvas.zones[] directly and absolute-positions each as a
    <div class="tp-zone kind-{kind} style-{style_variant}">.

  templates.normalized.json (overlay-mode templates):
    testimonial_overlay, product_overlay
    Declares placement_policy.element_priority_order — list of
    elements (logo, headline, product_meta, cta, quote) with
    region_hint and slot_binding. The renderer's placement service
    computes per-element rects at render time using
    OverlayZoneArtifact (brightness grid + restriction zones from
    Gemini Vision) and a greedy algorithm respecting
    subject_overlap_budget_pct. Text avoids the subject.

Both pipelines converge at the renderer's DOM layer: every zone
ends up as a tp-zone div with kind/style_variant classes that the
CSS in ads.html styles. Style bindings (panel_bg, headline_text_
color, cta_button_bg, etc.) always come from
templates.normalized.json.style_bindings and resolve via
templateRegistry.resolveStyleBindings → CSS variables on #tpStage.

Rule going forward (until the unification refactor below lands):

  New templates that AREN'T full-bleed overlays go canvas-first.
  Declare in canvas.v1.json with per-ratio zones[].
  Style bindings, validation, truncation rules go in
  templates.normalized.json.

  New full-bleed overlays (text-over-subject) keep
  placement_policy.element_priority_order in
  templates.normalized.json so subject-aware placement can adapt.

Backlogged: unification refactor that migrates overlay zone
declarations into canvas.v1.json with a placement_mode:"adaptive"
flag, strips zones[] from templates.normalized.json, and unifies
the two zone-loading paths in templatePreview.js. Triggered when
adding a 5th overlay-mode template, a renderer change touching
both paths, or a new contributor getting confused. See backlog.csv.


Stage 6b — Template binding resolution (services/templateRegistry.js)
---------------------------------------------------------------------
Each template (schemas/rsSocialProof.templates.normalized.json) declares:

  zones[]            ordered list of render zones (text, media, badge,
                     button, proof_bar, meta_block, text_card …)
  slot_adapter       per-zone input-path priority chain
  style_bindings     per-style-slot input-path priority chain + default
  truncation_rules   line / char caps per slot (max_lines, max_chars)
  validation         required_all_of, required_any_of, minCount per slot

Slot resolution (resolveStyleBindings, lines 190-214):
  For each binding entry:
    1. Walk source_priority array in order
    2. First non-null, non-empty value from input.* wins
    3. If all sources are empty, fall back to `default`
    4. If still empty, slot reads as null and the template's
       fallback styling kicks in

Contrast guard (applyContrastGuard, post-resolution):
  For pairs declared in CONTRAST_PAIRS (text → paired bg surface):
    - Compute WCAG luminance + contrast ratio
    - If ratio < 8.0 (AAA threshold) → override the text binding to
      white or brand fallback that DOES pass
  Catches "white headline on a light vibrant palette" and similar
  contrast failures that would otherwise ship.

Active V1 templates (both support ugc + product_image variants):

  testimonial_spotlight
    layout:          quote_focus base + rating_bar overlay
    zones:           hero_quote (social_proof.primary_quote.text |
                     copy.headline), support_media (product.hero_media
                     .video/image | creator.portrait_media |
                     product.image), rating_bar (proof_bar),
                     product_meta (name, price, currency, badges),
                     cta (button)
    style_bindings:  panel_bg ← media.palette_dominant
                     headline_text_color ← media.palette_vibrant
                       default #FFFFFF, contrast-guarded
                     cta_button_bg ← media.palette_vibrant |
                                       brand.accent_color | #FF6B35
                     font_family_headline ← brand.font_family
    truncation:      headline ≤3 lines, quote ≤4 lines, cta ≤24 chars
    validation:      required_any_of=[primary_quote.text | headline]
                     required_all_of=[product.name, cta.text]

  ugc_split_screen
    layout:          split_panel base (image + product side-by-side)
                     + rating_bar overlay
    zones:           section_header (copy.eyebrow | product.category |
                     brand.tagline), product_meta (text from
                     product.description | brand.summary |
                     brand.tagline), support_media (same priority as
                     spotlight), rating_bar, cta
    slot_adapter:    split-headline support
                     (copy.headline_lead + copy.headline_main)
    truncation:      headline ≤3 lines, product_description ≤12 lines,
                     cta ≤24 chars
    validation:      required_all_of=[product.name, cta.text]

  testimonial_overlay
    UGC-only (gated by TEMPLATE_SUPPORTS_VARIANT). Creator quote sits
    over a UGC scene, so it doesn't make sense on a catalog hero.

  product_overlay
    Both variants supported but not part of the V1 active pair in the
    smoke test.

Style bindings live in the per-template JSON spec, not in code.
Adding a new style slot:
  1. Add the binding to schemas/rsSocialProof.templates.normalized.json
  2. Add the slot to the renderer's CSS (frontend/app/src/ads-renderer/)
  3. If contrast-sensitive, add the (text, bg) pair to CONTRAST_PAIRS
     in services/contrastGuard.js


Ad lifecycle states
-------------------
queued     created by expandWizardJob; no render output; awaiting selection
rendering  selected by selectAdsForRun; Puppeteer in flight
draft      render succeeded; default ready-to-publish state
live       operator promoted via PATCH /api/ads/:id (status='live')
archived   operator soft-deleted via PATCH (status='archived')
failed     render failed; renderError on doc; no auto-retry

Operators only drive draft → live → archived (and back). The pipeline
manages queued / rendering / failed automatically.


Identity digest (queue-time dedup)
----------------------------------
identityDigest = sha256(JSON.stringify({
  campaignId, productId, mediaId, template, aspectRatio,
  variantKind, paletteSource, ctaText, ctaUrl, ctaUrlParams
}))

Computed in computeIdentityDigest (campaignAdsGenerationService.js).
Unique index on Ad: { campaignId: 1, identityDigest: 1 } unique.

  Same identity in the same campaign = same digest = unique-index
  rejects the duplicate insert.
  Different campaigns can have the same combo (intentional — separate
  operator action).

Removed the render-time derivationDigest dedup (it was based on
resolved copy and ran on every render). Identity-based queue-time
dedup is cheaper and earlier.


Cloudinary asset path
---------------------
folder:    ads/{brandId}/{campaignId}{isOverlay ? '/overlay' : ''}
publicId:  {ratio}-{template}-{shortMedia}-{shortDigest}{overlaySuffix}

  shortMedia  = String(mediaId).slice(-8)
  shortDigest = identityDigest.slice(0, 8)
  overlaySuffix = '-overlay' for the transparent video overlay PNG

overwrite: true — same identityDigest always lands at the same publicId,
so a re-render replaces (no orphans).

Operational limits
------------------
RENDER_CONCURRENCY             env, default 2    in-flight Puppeteer renders
MAX_CREATIVES_PER_RUN          env, default 6    cap per CampaignRun
SHIPPING_RATIOS                hardcoded         1:1, 9:16, 16:9
SUPPORTED_TEMPLATES            hardcoded         testimonial_spotlight,
                                                 ugc_split_screen,
                                                 testimonial_overlay,
                                                 product_overlay
BRAND_ONLY_MEDIA_LIMIT         25                brand-only seed pool
EXPANSION_PRODUCTS_PER_POST    env, default 25   top-K products per post in
                                                 category/brand expansion
MAX_ADS_PER_GENERATION_RUN     env, default 200  cartesian cap per run

Raising MAX_CREATIVES_PER_RUN: each run drains more queued ads but
takes proportionally longer. Bottleneck is Chromium memory at 2-3
parallel; bumping concurrency without bumping the Render instance
size will OOM.


Known gaps / things that will fail today
----------------------------------------
1. variantKind='product_image' ads render. assembleInput suppresses
   creator/ugc/performance blocks for this variant, and the prompt
   skips the SOCIAL CONTEXT section. testimonial_overlay is the only
   V1 template that doesn't support product_image (creator quote over
   a UGC photo only makes sense with real UGC).

2. Videos rarely render because readinessScore=null sorts them last.
   Drain images first; if you want a video, manually pick one via
   the Media Library's deep-link.

3. No "render more from this campaign" UI button. Workaround: hit
   Generate Ads again with the same picks — expandWizardJob is
   idempotent (unique index swallows dups), then selectAdsForRun
   picks the next-best queued ads.

4. Failed ads have no retry UI. Workaround: delete the failed ad,
   re-queue via Generate Ads. The unique index makes this clean.

5. CampaignRun.errors[] caps errors per run. Failed Ads also carry
   their own renderError so diagnostics survive on the doc.
